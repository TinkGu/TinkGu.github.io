Gitbook

发表于 2015-10-30

文艺码农的铜版纸

Gitbook

今天学习React的时候, 读到一份官方文档的中文版

阅读页面简洁大方, 有点类似我现在使用的这个博客主题, 怀疑是用了什么博客构建技术.
看了一下它的域名, 发现并不是xx.github.io, 而是xx.gitbooks.io. 我原以为gitbook是github推出的某项服务, 搜索了一下发现是一个在线电子书构建工具.

它的官网是GitBook.com.

转了一圈发现, 其各种操作逻辑甚至页面设计和github很相像, 主打在线协作共同写作, 比如在它的热门项目里, 有台北市长柯文哲用该工具写的书(噗). 总的还是以技术书籍居多, 比如《The Swift Programming Language》中文版.
倒不必被git这个字眼唬住, 实际上它提供了全PC平台的客户端, 就算不懂git只要会一点markdown语法就可以书写自己的电子书.

刚好我打算自行翻译英文版的JavaScript: The Good Parts, 放在博客里的话, 要是连续阅读章节, 或是跳读, 都不是很方便, 也懒得写专门的css… 整理成电子书阅读体验会更好.
不过说到最终让我决定用起GitBook的一点, 那就是它允许通过关联GitHub的仓库(repo), 只需要向GitHub的仓库提交代码, 就能够更新到GitBook对应的书籍中. 毕竟它们共用了一个git仓库嘛.
好处是push方便, 而且就算它提供了网页在线编写和客户端, 我还是喜欢用cmd markdown.

总结一下它的功能

  • 基于Git, 还能直接关联GitHub的某个仓库, 同步内容
  • 使用的是md语法, 仅就基础样式就很简洁方便
  • 直接生成在线网站, 相当于免费主页, 还支持下载.pdf, .epub, .mobi
  • 支持绑定其它域名
  • 提供了命令行工具
  • 还有不少的插件, 官方提供了dev-api

快速入门

GitBook可以通过npm安装命令行工具, 不过我刚摆弄这玩意儿的时候, 没有从官网的文档上找到命令行的命令介绍, 所以就通过使用网站完成了基础的操作.

  • 直接注册, 验证通过后, 在Your Profile中, 关联GitHub账号.
  • 选择新建书籍,在选项卡里选择关联事先在GitHub新建的仓库.
  • 之后, 就完成了GitHub的关联, 默认生成了如下文件

进阶

其实网上的快速入门教程不少, 不过我发现他们玩的GitBook怎么和我的有点不一样呢. 官方应该更新了下页面设计, 但我看了下教程里之前的版面, 反而是现在和GitHub更像了…
总之, 操作逻辑仍然不变, 前人的资料还可一看. 对我来说, 值得关注就在gitbook命令行操作和定制的md语法, 具体资料见最后一节. 在这里, 提下和资料不同的几点变化.

  • npm命令行工具从 npm install -g gitbook改为了npm install -g gitbook-cli. 如果装错, 需先卸载前者.

缺点

从国内访问GitBook比较慢, 我看到好多book都并不以xx.gitbooks.io开头, 而是移到了国内. 好在gitbook是支持绑定域名的, 所以速度敏感的话, 可以参考资料进行迁移. 也有教程是介绍发布到GitHub Pages的.

化蝶

发表于 2015-10-18

谷歌学派的行为艺术

学习JS有两大入门书, 都是O社出品的. O社专门做技术书籍, 封面大多为动物素描, 所以这一系列的书被戏称为动物书.
其中, JS权威指南的封面是一匹犀牛, JS语言精粹的封面是一只蝴蝶, 所以分别得到别名, 犀牛书和蝴蝶书.

一般来说, 犀牛书和蝴蝶书侧重各有不同. 前者较为全面, 后者则专注于剖析核心. 阅读蝴蝶书, 并配合如谷歌的Js style guide, 可以养成很不错的编码风格.

蝴蝶书本身很薄一小本, 我向图书馆借阅过它的英文影印版, 配合谷歌有道, 很快便能读完了. 之所以想到要翻译, 不是因为中文版翻得太渣, 大破新立, 而是洋泾浜四六级, 写个README.md都成问题. 翻译全算是个人练习.

是谁的眼神锁定我

因为是翻译着玩玩, 所以也不是更得很勤快. 原本连载在博客里, 后来用上了GitBook, 托管到了gitbooks.io.

喏, 戳

这边也贴一点译文.

当我还是一个初出茅庐的新手程序员时, 我曾想要学会我手头那些语言的每一个特性, 并且在编写代码时尽可能去用上所有这些特性. 我想要借此炫技, 而且确实做到了. 谁想知道某个特性要怎么用, 他只需来找我, 就能得到答案.
但最终, 我发现采用某些特性来解决问题将利大于弊. 它们要么缺乏规范, 定义不清, 要么很可能引发副作用. 它们有时导致代码变得难以阅读和维护, 有时导致我的代码风格变得过于复杂且容易出错. 有些特性本身就是设计错误, 因为语言设计者也难免有时犯错.

原文

When I was a young journeyman programmer, I would learn about every feature of the languages I was using, and I would attempt to use all of those features when I wrote. I suppose it was a way of showing off, and I suppose it worked because I was the guy you went to if you wanted to know how to use a particular feature.
Eventually I figured out that some of those features were more trouble than they were worth. Some of them were poorly specified, and so were more likely to cause portability problems. Some resulted in code that was difficult to read or modify. Some induced me to write in a manner that was too tricky and error-prone. And some of those features were design errors. Sometimes language designers make mistakes.
